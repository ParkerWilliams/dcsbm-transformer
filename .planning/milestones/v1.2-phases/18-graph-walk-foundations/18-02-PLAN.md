**1. Update `src/evaluation/behavioral.py`:**

```python
class RuleOutcome(IntEnum):
    """Outcome of rule compliance check at a generation step.

    UNCONSTRAINED: No jumper rule active at this step.
    PENDING: A jumper rule is active but its deadline has not yet been reached.
    FOLLOWED: Jumper rule deadline reached, correct block arrived.
    VIOLATED: Jumper rule deadline reached, wrong block arrived.
    """
    UNCONSTRAINED = 0
    PENDING = 1
    FOLLOWED = 2
    VIOLATED = 3
```

Update the module docstring to reference 4 classes.

Update `classify_steps` function:
- Change initialization from `RuleOutcome.NOT_APPLICABLE` to `RuleOutcome.UNCONSTRAINED`
- After checking all deadlines at each step, if no deadline resolved BUT there are active constraints with remaining steps, mark the step as `PENDING`
- Specifically: after the existing deadline-check loop, add logic to check if any active_constraint has deadline > t+1 (i.e., still waiting). If yes AND the step was not already marked FOLLOWED or VIOLATED, set `rule_outcome[b, t] = RuleOutcome.PENDING`.
- Keep the `break` after the first deadline resolution (only one constraint resolves per step).

The key logic change in the inner loop of classify_steps:
```python
# After checking all deadlines for step t:
# If no deadline matched, check if any constraint is still pending
if rule_outcome[b, t] == RuleOutcome.UNCONSTRAINED:
    for deadline, target_block in active_constraints:
        if deadline > t + 1:  # Constraint not yet resolved
            rule_outcome[b, t] = RuleOutcome.PENDING
            break
```

Note: The PENDING label applies to the transition at step t (the pair u->v). A step is PENDING if a jumper constraint exists but its deadline is in the future. This gives downstream analysis the ability to distinguish "no constraint was relevant" from "a constraint was active but not yet due."
